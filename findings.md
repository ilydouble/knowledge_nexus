# 发现与决策

## 需求
- 用户希望基于 Pi-Agent 梳理调研材料，逐步构建自有本体和知识图谱。
- 项目形态不是一次性脚本，而是一个可交互的智能体客户端/操作系统。
- 需要形成工具、技能、记忆等长期能力，支持后续交互式分析数据、初始化本体、构建图谱和问答。
- 未来成果需要导入 KGraph，用于项目交付和进一步细化；Pi-Agent 侧直接构建图谱和问答也成立，不与 KGraph 交付矛盾。

## 研究发现
- 现有仓库已经具备 Knowledge OS 架构：Cloudreve 文件底座、候选抽取与审核、Neo4j 图谱、Postgres 证据、MCP Server、Web 控制台。
- 当前语义管道支持文档分类、LLM 抽取、候选批次、预览、提交和图谱问答，适合承接“材料到图谱候选”的闭环。
- 调研材料把园区数据归纳为三类数据需求：结构化关系与空间数据、非结构化文档与静态台账、动态时序与事件数据。
- 调研材料进一步提出四大知识层级：动态感知层、物理骨架层、业务经验层、顶层标准层。
- 顶层标准建议使用 Brick Schema 和 RealEstateCore；研究背书包括建筑能效知识图谱综述、The World Avatar、FDD 与控制融合论文方向。
- 可利用的数据源包括 EMS/BMS/IoT/消防安防/空间台账/设备台账/工单，以及公开数据集 ASHRAE、SHIFDR、AHU FDD、LBNL AFDD。

## 技术决策
| 决策 | 理由 |
|------|------|
| 以园区四层知识作为领域本体主轴 | 与用户材料一致，便于解释、抽取和交付 |
| 将 Brick/REC 映射为 L1 标准层，本项目自定义园区概念作为 L2 项目层 | 兼顾国际标准背书和本地项目可落地性 |
| 保留候选审核机制 | 园区图谱涉及交付质量，不能让抽取结果直接污染正式图谱 |
| 设计 Pi-Agent 工具、技能、记忆三类资产 | 工具负责动作，技能负责流程，记忆负责项目长期上下文 |

## 遇到的问题
| 问题 | 解决方案 |
|------|---------|
| KGraph 导入格式尚未明确 | 暂按中立 YAML/JSON 本体种子设计，后续增加导出适配器 |
| 第一版优先级尚未明确 | 先给出推荐架构与最小闭环，再让用户确认实施顺序 |
