Knowledge Nexus Architecture

整体架构设计与当前实现阶段

Knowledge Nexus 以 Cloudreve 为物理文件底座，以 Neo4j 知识图谱为语义叠加层，通过 MCP 协议将图谱开放给 Claude Code 等 AI 智能体。向量嵌入按需启用，当前以"图谱优先"模式运行。

1. 架构核心命题

本项目的核心原则是：物理存储受控，逻辑链接自由。原始文件、下载、预览和硬权限仍由 Cloudreve 管理；摘要、标签、实体、图谱关系和 GraphRAG 上下文由 Nexus 作为旁路语义层负责，并通过 MCP 协议暴露给 AI 智能体。

Cloudreve
文件、目录、权限、全文检索
→
Worker + Pipeline
分类、解析、LLM 提取
→
Neo4j + Postgres
图谱 + 元数据

MCP Server 在图谱与 AI 智能体之间搭桥，Claude Code 可直接调用图谱查询工具。

2. 分层设计

L1 顶层知识层

承载标准本体、实体类型、受控词表和全组织一致的知识定义。它是系统里的“秩序层”，未来应由管理员或领域专家维护。

Standard Ontology 全组织可见

L2 团队知识层

承载团队、项目、部门形成的共识图谱。它连接文件、项目、交付物、人员和经验沉淀，是协作知识库的主体。

Collective Domain 团队可见

L3 个人认知层

承载个人笔记、私人标签、主观联想和 [[Link]] 式双向链接。默认仅本人可见，主动发布后才进入团队知识层。

Personal Cognition 个人私有

3. 主要组件职责

组件 当前职责 代码位置
Worker 三循环并发：① SSE 监听 Cloudreve 文件事件；② 每 10 分钟全盘扫描，发现未处理文件；③ 每 30 秒批量处理队列（每批 3 个），防止 LLM API 并发爆炸。 nexus/worker.py
SemanticPipeline 文件处理主流程：FileGate 过滤 → ContentParser 解析 → DocumentClassifier 分类 → KnowledgeExtractor 提取 → Neo4j/Postgres 入库。可选向量嵌入（VECTOR_BACKEND=milvus）。 nexus/services/pipeline.py
FileGate 白名单过滤，只处理 PDF/DOCX/XLSX/TXT/MD/CSV/JSON 等有效文档格式，跳过图片、视频、压缩包等噪声文件。 nexus/services/file_gate.py
ContentParser 多格式解析：PDF（pdfplumber）、DOCX（python-docx）、Excel（openpyxl，只读结构摘要）、纯文本（1000 字/块）。 nexus/services/content_parser.py
DocumentClassifier 7 类文档自动分类（academic_paper / technical_doc / meeting_minutes / report / contract / email / tabular_data / general），优先级：file_type → 扩展名 → 文件名关键词 → 内容关键词 → 兜底。 nexus/services/document_classifier.py
KnowledgeExtractor 富本体提取：每种文档类型有专属实体类型和关系约束，注入 prompt 后调用 ZhipuAI LLM。长文档（>10,000 字）走 Map-Reduce 分段提取后合并去重。 nexus/services/knowledge_extractor.py
Neo4j Graph Store 以 MERGE 写入文件节点、实体节点和关系边（幂等，重新扫描不产生重复）。支持全图查询、一跳邻域、实体搜索和文档子图。 nexus/graph/neo4j_store.py
MCP Server 通过 FastMCP 暴露 5 个工具：list_documents、get_document、search_entities、get_document_graph、find_documents_by_tag，供 Claude Code 等 AI 智能体查询图谱。 nexus/mcp_server.py
FastAPI Gateway 暴露健康检查、Cloudreve OAuth、录入同步、任务状态、文档列表、图谱邻域、触发扫描等接口。 nexus/app_factory.py
Web Console 文件处理状态、图谱可视化、GraphRAG 问答演示界面。 apps/web/

4. 数据与调用流程

  1. Cloudreve 文件上传触发 SSE 事件，Worker 接收后创建 IngestionJob（pending 状态）。周期扫描也会为未处理文件补充创建任务。
  2. 批处理循环取出 pending 任务，调用 SemanticPipeline.process_file()：FileGate 过滤 → ContentParser 解析 → DocumentClassifier 分类 → KnowledgeExtractor 提取实体/关系/摘要/标签。
  3. 提取结果写入 Neo4j（实体图谱，MERGE 幂等）和 Postgres（文档摘要 + 标签 + chunks）。
  4. 任务状态更新为 succeeded 或 failed（记录错误阶段）。
  5. AI 智能体通过 MCP Server 查询 Neo4j 图谱和 Postgres 文档元数据，无需直接操作数据库。
  6. 如开启 VECTOR_BACKEND=milvus，Pipeline 在入库后额外批量嵌入文本块写入 Milvus，支持向量相似检索。

6. 当前部署视图

推荐以 Host 模式运行：复用已有的 kg-* Docker 基础设施（Postgres、Neo4j、Redis），按需启动以下 4 个进程。

Host 开发模式（推荐）

  • uvicorn apps.api.main:app --reload → API :8000
  • cd apps/web && npm run dev → Web :5173
  • python -m nexus.worker → 文件处理 Worker
  • python -m nexus.mcp_server → MCP Server（供 Claude Code）

向量嵌入（按需开启）

  • 默认 VECTOR_BACKEND=none，Milvus 不连接，无嵌入 API 调用。
  • 改为 VECTOR_BACKEND=milvus 后重启 API + Worker 即可启用。
  • 历史文件不自动补跑，可手动重新触发扫描（幂等，Neo4j 图谱不重复）。

7. 下一阶段建议

  1. 接入外部 GraphRAG 平台（Dify / LangChain / LlamaIndex），将 Neo4j 图谱作为检索源，实现问答闭环。
  2. 实现 Experience Replay：Agent 运行时发现的细粒度知识，异步写回 Neo4j 图谱，形成持续学习闭环。
  3. 按需开启向量嵌入（VECTOR_BACKEND=milvus），形成"图谱 + 向量"混合检索，提升开放域问答精度。
  4. 扩展 MCP Server 工具集：增加实体路径查找、多跳推理、图谱写回（Experience Distillation）等工具。
  5. 扩展 Web Console：图谱层级开关、文档处理历史、实体详情页、GraphRAG 问答界面。
  6. 将 L1 本体、L2 团队图谱的治理流程落到数据模型和管理 API 上。
